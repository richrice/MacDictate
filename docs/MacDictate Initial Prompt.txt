Build a complete native macOS application named **MacDictate**.

MacDictate is a menu-bar push-to-talk dictation utility designed for entering natural-language prompts into terminal-based coding agents such as Codex CLI and Claude Code.

Do not merely describe the implementation. Create the project, source files, tests, build scripts, documentation, and a working application.

## Primary workflow

The expected workflow is:

1. The user focuses Terminal, iTerm2, VS Code, Cursor, or another text-entry application.
2. The user presses and holds a global hotkey.
3. MacDictate begins recording the microphone without stealing focus.
4. The user speaks a coding instruction.
5. The user releases the hotkey.
6. MacDictate stops recording and sends the audio to the OpenAI transcription API.
7. MacDictate inserts the returned text at the current cursor position in the previously focused application.
8. The user reviews the text and manually presses Return.

The app must never automatically submit the prompt or simulate the Return key.

## Technology requirements

Build the application using:

* Swift
* SwiftUI for settings and normal UI
* AppKit where required for menu-bar, floating panels, application focus, and paste handling
* AVFoundation for microphone recording
* Security framework for Keychain storage
* Accessibility APIs and Core Graphics events for text insertion
* URLSession for OpenAI API requests
* XCTest for automated tests

Target macOS 14 or newer.

Do not use Electron, Node.js, Python, a local web server, or a browser wrapper.

Avoid third-party dependencies unless they clearly improve reliability. Prefer native Apple frameworks. Explain any dependency before adding it.

Use Swift concurrency with `async`/`await`. Keep UI work on `@MainActor`. Enable strict concurrency checking and resolve concurrency warnings rather than suppressing them.

A new recording must not begin while the previous recording is being transcribed or inserted.

## Global push-to-talk hotkey

The default hotkey should be:

`Option + Space`

Required behavior:

* Key down starts recording.
* Key up stops recording and begins transcription.
* Auto-repeat must not create additional recording sessions.
* A brief accidental press under 250 milliseconds should cancel instead of sending nearly empty audio.
* The hotkey must work while another application is focused.
* The application must not steal focus when the hotkey is used.
* The underlying foreground application must not receive the hotkey keystroke.

Prefer a native global-hotkey implementation that provides both key-down and key-up events.

Use Carbon `RegisterEventHotKey` if it remains the most reliable native approach for press-and-release behavior. Isolate any legacy API usage inside `GlobalHotkeyManager` so it can later be replaced.

Make the hotkey configurable in Settings after the basic fixed-hotkey version works. Detect and report registration conflicts.

## Microphone recording

Use `AVAudioEngine` to capture microphone audio.

Record to a temporary WAV file suitable for the OpenAI transcription API.

Preferred format:

* mono
* 16 kHz when practical
* 16-bit linear PCM
* WAV container

Use `AVAudioConverter` when microphone hardware produces a different sample rate or channel count.

Requirements:

* Request microphone permission clearly.
* Include a meaningful `NSMicrophoneUsageDescription`.
* Handle missing input devices.
* Handle device changes and recording interruptions.
* Display the selected input device in Settings.
* Allow the user to choose among available microphone input devices when feasible.
* Delete temporary audio files immediately after success, cancellation, or terminal failure.
* Do not retain an audio history by default.
* Stop recordings automatically after five minutes.
* Reject an empty or effectively silent recording with a friendly message.
* Keep generated uploads comfortably below the API upload-size limit.

Do not log raw audio data.

## OpenAI API authentication

Allow the user to enter an OpenAI API key in Settings.

Store it only in macOS Keychain using the Security framework.

Requirements:

* Never store the key in `UserDefaults`.
* Never write the key to a source file, plist, log, crash message, or environment dump.
* Show the key as a secure field.
* Provide Save, Replace, and Delete actions.
* Display only a masked indication that a key is configured.
* Keep the Keychain service and account identifiers centralized as constants.
* Handle duplicate-item and item-not-found Keychain results correctly.
* Avoid keeping the key in memory longer than needed for a request.
* Redact authorization headers and secrets from all logging.

Add a README warning that users must never commit API keys to Git.

## OpenAI transcription request

Use:

`POST https://api.openai.com/v1/audio/transcriptions`

Send a standards-compliant `multipart/form-data` request containing:

* `file`: the WAV recording
* `model`: selected transcription model
* `response_format`: `text`
* `prompt`: the configured transcription context
* `language`: `en`, when the user has selected English rather than automatic language detection

Default model:

`gpt-4o-mini-transcribe`

Also allow:

`gpt-4o-transcribe`

The model selection must be configurable in Settings. Do not hard-code model behavior throughout the app; centralize supported models.

Create the multipart request in a dedicated, unit-tested builder. Use binary-safe `Data` construction and a randomly generated boundary.

Use a reasonable request timeout, such as 60 seconds.

Handle:

* invalid API key
* insufficient API balance or quota
* rate limiting
* network unavailable
* DNS and TLS failures
* server errors
* malformed responses
* empty transcription
* request cancellation

For HTTP 429 and transient 5xx errors, perform at most one automatic retry with a short exponential delay. Do not retry authentication or other permanent 4xx errors.

Parse both plain-text responses and JSON error bodies.

Provide user-friendly errors without hiding the underlying technical reason. Technical details may be available through a Copy Details button, but secrets must remain redacted.

## Developer vocabulary prompt

Provide this editable default transcription-context prompt:

“Transcribe an English software-engineering instruction accurately. Preserve technical names, filenames, file paths, shell commands, command-line flags, URLs, version numbers, variable names, class names, acronyms, and quoted text. Relevant vocabulary includes Codex, Claude Code, Git, GitHub, Swift, SwiftUI, Xcode, Kotlin, Java, TypeScript, JavaScript, Python, Docker, npm, pnpm, Homebrew, Jira, SVN, ESP32, STM32, macOS, iOS, Android, API, JSON, YAML, SQL, SQLite, OpenRouter, Traxxas, and Firebase. Use normal punctuation and capitalization.”

Allow the user to modify and reset this prompt in Settings.

Do not falsely imply that the prompt guarantees exact spelling. It is context supplied to the transcription model.

## Text insertion

The transcript must be inserted into the application that was focused when recording began.

Capture and retain the frontmost application identity at the beginning of dictation.

Implement text insertion using this order:

1. Attempt focused-element insertion using the macOS Accessibility API.
2. When direct Accessibility insertion is unsupported or fails, use a clipboard-and-paste fallback.
3. When Accessibility permission is unavailable, copy the transcript to the clipboard and notify the user to paste manually.

For Accessibility insertion:

* Use `AXUIElementCreateSystemWide`.
* Obtain the focused UI element.
* Attempt insertion through the appropriate selected-text attribute.
* Preserve existing selected-text behavior where possible.
* Handle applications that reject setting accessibility attributes.

For clipboard fallback:

1. Snapshot all existing pasteboard items and their available representations.
2. Put the transcript on the general pasteboard.
3. Generate a Command-V key event using `CGEvent`.
4. Wait long enough for the foreground application to consume the paste.
5. Restore the previous clipboard only when the pasteboard has not subsequently been changed by the user or another application.

Do not overwrite a newer user clipboard value during restoration.

Do not simulate Return, Enter, or any command that would submit the prompt.

Test text insertion in at least:

* Apple Terminal
* iTerm2 when available
* VS Code
* a standard SwiftUI `TextEditor`
* a browser text area

Document any application-specific limitations.
